# A. STARTUP CANVAS (CORE)

## 1. The Problem (real, structural)

**Story framing (how it actually happens):**

> A decision needs to be made.
> Insurance is required.
> A document is sent.
> Everyone pauses.

Not because insurance is missing —
but because **verification is slow, manual, and risky**.

### Root causes

* Insurance documents:

  * differ by country
  * differ by insurer
  * differ by language
* Verification is:

  * PDF-based
  * human-reviewed
  * liability-heavy (“what if I approve the wrong thing?”)

### Result

* Delays
* Over-verification
* Escalations
* People default to **inaction**

This happens in:

* education
* mobility
* contracting
* logistics
* healthcare access
* temporary cross-border work

This is **not a niche problem**.

---

## 2. The Insight (your edge)

> Insurance is not a document problem.
> It’s a **trust interface problem**.

Institutions don’t want policies.
They want **answers**:

* Is it valid?
* Is it compliant?
* Is it active **right now**?

They want a **binary signal**, not interpretation.

---

## 3. The Solution (clearly scoped)

> A universal insurance verification layer that turns insurance into a real-time compliance signal.

### What you provide

* A **standardized Insurance Passport**
* A **rules-based verification engine**
* A **single scan → single answer**

### What you explicitly do NOT do

* You don’t underwrite
* You don’t replace insurers
* You don’t handle claims

That keeps you:

* unregulated (or lightly regulated)
* scalable
* enterprise-friendly

### Core object: Insurance Passport

A normalized representation of an insurance policy.

**Contains**

* Policy identifier
* Issuer
* Coverage attributes
* Jurisdiction
* Validity window
* Compliance flags (liability, repatriation, etc.)

**Form**

* Human-readable (1 page)
* Machine-readable (JSON)
* QR code pointing to live status

---

### Verification Engine

Rules engine that answers one question:

> “Does this insurance satisfy *this* requirement *right now*?”

Outputs:

* ✔ Verified
* ✖ Not verified (+ reason)

No gray zones.

---

### Verification Interface

* QR scan
* API call
* Dashboard lookup

This is where adoption happens.
